# Public Static Site to `apps/sis-next` Migration Plan

Date: 2026-05-29

Status: Planning document

Target app: `apps/sis-next`

Source app: `public`

Backend to retire: `functions`

## 1. Executive Summary

Migrate the Student Information System from the static Firebase Hosting site in `public/` into the Next.js application in `apps/sis-next/`. The Next app becomes the single owner of UI, authenticated SSR, API routes, server-side Firestore access, user management, student mutations, import/export workflows, and deployment.

The desired end state is:

- `apps/sis-next` is the only active web application.
- Firebase Authentication is integrated with SSR using Firebase Server Apps for user-context Firebase SDK reads.
- Next Route Handlers and server-only modules replace the callable functions currently deployed from `functions/`.
- The root `functions` deployment is removed from `firebase.json`.
- Static `public/*.html`, `public/js/*.js`, and static-site-only Firebase runtime code are archived or deleted after parity is verified.
- Firebase App Hosting is the preferred production target because Firebase's own Next.js Hosting docs state that dynamic server logic deployed through framework-aware Firebase Hosting is translated into Cloud Functions. That conflicts with the goal of moving backend behavior out of `functions/`.

Official references checked on 2026-05-29:

- Firebase SSR with `FirebaseServerApp`: https://firebase.google.com/docs/web/ssr-apps
- Firebase Hosting framework-aware Next.js behavior: https://firebase.google.com/docs/hosting/frameworks/nextjs
- Firebase App Hosting setup: https://firebase.google.com/docs/app-hosting/get-started

## 3. Architecture Decision

### Decision A: Use Firebase App Hosting for Production

Use Firebase App Hosting for `apps/sis-next`.

Reason:

- The user wants backend behavior inside the Next.js backend, not in `functions/`.
- Firebase's framework-aware Hosting path for Next.js deploys dynamic server logic to Cloud Functions. That is acceptable for some apps, but it blurs the line with the current `functions/` backend and does not match the stated target.
- `apps/sis-next/apphosting.yaml` already exists, which suggests this repo is already pointed toward App Hosting.

Production target:

- Firebase App Hosting backend with root directory `apps/sis-next`.
- Runtime command from `apphosting.yaml`: `npm run start`.
- Use managed Application Default Credentials in production.
- Keep Firestore rules/indexes deployed from the repo root unless Firebase App Hosting project layout is later reorganized.

Local target:

- `npm --prefix apps/sis-next run dev`
- Firebase Emulator Suite for Auth and Firestore.
- No Functions emulator after Route Handler migration is complete.

### Decision B: Use Both Firebase Server Apps and Admin SDK

Use Firebase Server Apps for SSR reads that should execute with the signed-in user's Firebase Auth context.

Use Firebase Admin SDK for privileged backend-only operations:

- creating, updating, deleting Auth users
- setting and reading custom claims
- admin-only writes to `students` and `trash`
- transactional trash/restore/delete flows
- migration and seed operations

Reason:

- `FirebaseServerApp` is intended for SSR user-context Firebase SDK operations.
- The Admin SDK bypasses Firestore Security Rules, so it must stay inside server-only modules and Route Handlers.
- Some workflows require admin privileges beyond the current user's Firestore rules, especially user management.

### Decision C: Replace Callable Functions With HTTP Route Handlers

Do not keep the callable-function client contract.

Instead, expose typed internal HTTP endpoints under `apps/sis-next/app/api/*` and call them through a local client API wrapper.

Reason:

- Route Handlers are natural Next backend units.
- They are easier to test, compose, validate, cache-control, and deploy with App Hosting.
- They avoid requiring `firebase-functions` and callable protocol semantics inside the Next app.

## 5. SSR Authentication Design

### 5.1 Cookie and Token Strategy

Keep the existing session cookie route, but strengthen it:

- Cookie name: `__session`
- Cookie type: Firebase Admin session cookie
- Flags: `httpOnly`, `secure` in production, `sameSite: "lax"`, `path: "/"`
- Expiration: keep current 5 days unless product requirements need shorter sessions.
- Verify with `verifySessionCookie(sessionCookie, true)` so revoked sessions are rejected.

Add a Firebase Server App token bridge:

- Register a Firebase Auth service worker in the browser.
- The service worker refreshes the current Firebase ID token and attaches it to SSR-triggering requests, for example as `Authorization: Bearer <idToken>` or `x-firebase-auth-token`.
- In Server Components and Route Handlers that need user-context Firebase SDK reads, read the token from `headers()`.
- Initialize `FirebaseServerApp` with:
  - same public Firebase config as the client app
  - `authIdToken`
  - `releaseOnDeref` set to the request headers object
- Use `getAuth(serverApp)` and Firestore client SDK from that server app for user-context SSR reads.

Why not store the ID token as a long-lived cookie:

- Firebase ID tokens are short-lived.
- A non-httpOnly ID token cookie increases XSS blast radius.
- An httpOnly ID token cookie cannot be refreshed directly by the client.
- The official Firebase SSR docs recommend a service-worker token bridge for SSR token transmission.

### 5.2 Request Auth Helpers

Create `app/lib/server/auth.ts`:

- `getSessionUser()`
  - reads `__session`
  - verifies session cookie with Admin SDK
  - returns normalized `{ uid, email, name, role, claims }`
- `requireSessionUser()`
  - throws/redirects on missing session
- `requireAdmin()`
  - rejects unless `role === "admin"` or `admin === true`
- `getBearerIdTokenFromHeaders()`
  - reads the service-worker-provided ID token
- `getFirebaseServerAuth()`
  - initializes Firebase Server App from current headers and token
- `isAdminClaims()`, `isAppUserClaims()`
  - shared role checks

### 5.3 SSR Guarding

Move route guarding from client-only `AuthGuard` to server-first flows:

- `app/page.tsx` becomes an async Server Component.
- It calls `getSessionUser()`.
- If missing session, `redirect("/login")`.
- If present, load initial data server-side and pass it into a client shell.
- Keep client AuthProvider only for live sign-out, display state, and client actions.

Expected improvement:

- No "Checking your session..." flash on initial protected page load.
- SSR can render the correct role-gated UI on first response.
- Client code no longer decides whether protected content is accessible.

### 5.4 Login Flow

Keep email/password login initially:

1. Browser signs in with Firebase client SDK.
2. Browser gets fresh ID token with `getIdToken(true)`.
3. Browser POSTs token to `/api/auth/session`.
4. Route Handler verifies ID token with Admin SDK.
5. Route Handler creates `__session`.
6. Browser refreshes or navigates to `/`.
7. Service worker handles ID token attachment for SSR requests after registration.

Enhancements:

- Add `onIdTokenChanged` listener to refresh backend session after token refresh or claim changes.
- After admin changes a user's role, force the target user to refresh their token or sign in again.
- Add `POST /api/auth/session/refresh` only if the existing POST route becomes too broad.

## 6. Backend API Replacement Matrix

### 6.1 User Management

```text
GET    /api/users

```

Authorization:

- All endpoints require admin.
- Self-delete must remain blocked.

Implementation:

- Use Admin Auth SDK only.
- Reuse error mapping from `functions/src/index.ts`.
- Normalize role to `"admin"` or `"user"`.

  - `role`

- Admin can create a user.
- Admin can list users sorted by email.
- Admin can update display name and/or password.
- Admin can delete another user.
- Admin cannot delete self.
- Non-admin receives 403.
- Signed-out user receives 401.

### 6.2 Student Records

Replace callable functions:

- `saveStudentByAdmin`
- `updateStudentByAdmin`
- `moveStudentToTrashByAdmin`
- `restoreStudentByAdmin`
- `deleteTrashStudentByAdmin`

Route Handlers:

```text

PATCH  /api/students/[studentId]
DELETE /api/students/[studentId]

GET    /api/trash
```

Authorization:

- Read `students`: admin or user.

- Read `trash`: admin.

Implementation:

- Use Admin Firestore for mutations.
- Use either FirebaseServerApp Firestore or Admin Firestore for reads:
  - Prefer FirebaseServerApp for SSR user-context reads.
  - Use Admin for route handlers after explicit session/role checks.
- Preserve collection names: `students`, `trash`.
- Preserve document IDs as `student_id`.

Acceptance:

- Student IDs continue as `STU###`.
- Create sets `created_at` and clears `deleted_at`.
- Update preserves existing `created_at`.
- Move to trash is transactional.
- Restore is transactional.
- Permanent trash delete deletes only from `trash`.
- Non-admin cannot mutate students or trash.

### 6.5 Import and Seed Workflows

- `seedFirestoreFromBundledJson()` is available from client code.

Target Route Handlers:

```text

```

Authorization:

- `import/batch-options`: admin.
- `seed/firestore`: admin plus environment guard.

Environment guard for seed:

- Allow in local emulator/dev by default.

- Log the actor UID and counts.

Implementation:

- Prefer importing JSON server-side when possible instead of fetching public URLs.
- Remove `seedLocalFromBundledJson` from production UI. If a local demo mode is still needed, isolate it behind a dev-only flag.

Acceptance:

- Batch options import is idempotent.
- Seed route refuses production unless explicitly enabled.
- No unauthenticated user can seed Firestore.

Acceptance:

- Excel `J25` total remains correct.
- Downloaded ZIP names match current naming.

## 9. Firestore Rules Strategy

Phase 1, parity:

- Keep current Firestore rules mostly unchanged.
- Students and trash remain server-written only.
- Options and payout records can remain app-user client writable until all UI paths use Route Handlers.

Phase 2, backend-owned writes:

- Tighten all write paths so writes happen through Next backend.
- Suggested future rules:
  - `students`: app users read, no client writes
  - `trash`: admins read, no client writes
  - `payoutRecords`: app users read, no client writes
  - setup collections: app users read, admin write or no client writes depending on policy

Important:

- If SSR reads use FirebaseServerApp with user context, Firestore rules must continue to permit the relevant user reads.
- Admin SDK writes bypass rules, so Route Handler checks must be complete and tested.

## 12. Rollback Plan

Keep rollback simple until cutover is proven:

1. Keep root `public/` and `functions/` intact until Phase 6.
2. Keep current Firebase Hosting static deploy available as a legacy target or previous release.
3. Do not delete callable functions until Next Route Handlers have production parity.
4. If App Hosting rollout fails, route traffic back to legacy static hosting while preserving Firestore/Auth data.
5. Avoid schema-breaking Firestore changes during migration.

Rollback trigger examples:

- admin cannot create/update student records
- session cookie verification fails for valid users
- route handlers produce data corruption
- payroll export output differs materially from legacy output

## 13. Risks and Mitigations

### Risk: FirebaseServerApp Token Is Missing on First SSR Request

Mitigation:

- Use `__session` cookie for coarse SSR auth and redirects.
- Use service worker token bridge for user-context Firebase Server App reads.
- Fallback to Admin SDK reads only after explicit session authorization checks.

### Risk: Admin SDK Accidentally Reaches Client Bundle

Mitigation:

- Add `import "server-only";` to every server-only module.
- Keep server modules under `app/lib/server`.
- Do not import server modules from `"use client"` files.
- Verify with `npm --prefix apps/sis-next run build`.

### Risk: Firestore Rules Drift From Route Handler Authorization

Mitigation:

- Centralize roles in `app/lib/shared/roles.ts`.
- Test route authorization separately from Firestore rules.
- Re-audit rules after tightening client writes.

### Risk: Feature Parity Gaps From Static App

Mitigation:

- Use the exported function list in `public/js/store.js` as the parity checklist.
- Port one workflow at a time and test before deleting legacy code.
- Keep static site available until sign-off.

### Risk: Payroll Export Memory Usage

Mitigation:

- Start with existing client-side export if server memory is uncertain.
- If server-side, test large selections under App Hosting memory limits.
- Increase `memoryMiB` to `1024` if needed.
